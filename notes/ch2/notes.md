# ch2 실험 가이드 — exp02 전체 파인튜닝

> exp01과 코드는 동일, 옵션만 다르다. 이 문서는 실험 **전에** 예상을 적고,
> **후에** 결과와 비교하는 실험 노트다. (예상 → 실측 → 해석 습관 들이기 —
> 테슬라도 새 모델을 내보내기 전에 예상 지표를 먼저 걸어두고 실측과 비교한다.)

## 실행

```powershell
# 본 실험 (10에폭, lr 1e-4)
python src/train.py --name exp02_full_finetune --lr 1e-4

# 끝나면 대시보드 반영
git add docs/metrics && git commit -m "exp02 full finetune results" && git push
```

`--freeze-backbone`을 빼는 것과 lr을 1e-4로 낮추는 것, 차이는 이 둘뿐이다.

- 최종 test_acc: _59__% (exp01 69.4% 대비)
- 과적합 신호(test_loss 상승)가 나타날까? 몇 에폭쯤? _?__

## 결과 기록 (실측 후 채우기)

2026-07-11 실행. 로그: 최고 test acc **75.65%** (10에폭), 체크포인트 `checkpoints/exp02_full_finetune_best.pt`.

| 항목 | 예상 | 실측 | 해석 |
|---|---|---|---|
| 에폭당 시간 | 5~10배 (90~180초) | **~28초 = 1.5배** (첫 에폭만 76초 — 디스크 캐시 워밍업) | 늘어난 건 역전파 뒷부분과 옵티마이저뿐. 순전파·데이터 로딩은 exp01과 동일 → [q07](q07-why-slower-epochs.md)에 정리 |
| 에폭 1 test_acc | 낮다 (exp01 54.6%보다) | **62.4% — 오히려 높음** | lr 1e-4가 충분히 작아서 "초반에 백본이 흔들리는" 일이 없었다. 백본이 움직이자마자 fc만 학습할 때보다 잘 맞춤 |
| 최종/최고 test_acc | 59% | **75.65% (10에폭)** | 예상보다 +16.7%p. 백본 해동이 exp01 대비 +6.2%p. 다만 목표 80%에는 -4.4%p |
| test_loss 최저점 에폭 | ? | 10에폭 (0.860) — 상승 반전 없음 | 10에폭 안에서는 명확한 과적합 경고(test_loss 상승)가 안 나옴. 더 돌릴 여지가 있다는 뜻이기도 |

- 학습 곡선에서 눈에 띈 것: test_acc는 5에폭(75.4%)부터 75% 언저리에서 **정체**하는데
  train_acc는 계속 올라 80.5%까지 감 — train-test 간격이 벌어지는 건 "외우기 시작"하는
  신호다. 단 test_loss가 아직 내려가는 중이라 과적합 '확정'은 아님 (concepts.md의
  곡선 읽는 법 1번 항목).
- exp01 곡선과 겹쳐 보고 알게 된 것: exp02가 **1에폭부터 끝까지 전 구간 위**에 있다.
  exp01의 최종값(69.4%)을 exp02는 2에폭 만에 넘었다. "백본까지 미세조정하면
  같은 데이터로도 표현력이 확 달라진다"가 곡선 한 장으로 보임.
- 예상 복기: 에폭 시간(5~10배 예상 → 1.5배)과 초반 test_acc(낮을 것 → 더 높음) 둘 다
  빗나갔는데, 둘 다 "백본이 학습에 참여하면 비용/불안정이 클 것"이라는 같은 직관에서
  나온 오답이었다. 실제로는 lr을 10배 낮춘 것이 그 비용을 거의 다 흡수했다.

## 비교 실험 — q03을 곡선으로 확인

본 실험이 끝나면, lr만 10배 올린 쌍둥이 실험으로
[q03](../ch1/q03-full-finetune-low-lr.md)의 해설을 검증한다:

```powershell
python src/train.py --name exp02b_full_lr1e-3 --lr 1e-3
```

예상 (q03 해설이 맞다면): 초반 에폭에서 test_acc가 exp02보다 크게 낮거나
심하면 exp01 베이스라인(69.4%) 아래로 꺼지고, 끝까지 회복 못 할 것.
대시보드에서 세 곡선(exp01, exp02, exp02b)을 겹쳐 보면 "학습률이 사전학습
가중치를 지키는 안전장치"라는 게 그림 한 장으로 보인다.

**결과 (2026-07-11 실측): 예상 그대로, 예상보다 심하게.**

| | exp01 (fc만, 1e-3) | exp02 (전체, 1e-4) | exp02b (전체, 1e-3) |
|---|---|---|---|
| 에폭 1 test_acc | 54.6% | 62.4% | **21.7%** |
| 최고 test_acc | 69.4% | 75.7% | **55.0%** (10에폭에도 미회복) |

- 1에폭 만에 21.7% — 백본이 "이미지 보는 능력"을 첫 에폭부터 뭉개졌다는 뜻.
  이후 10에폭 내내 exp01 베이스라인 근처도 못 갔다 (catastrophic forgetting을
  4,000장으로 복구 못 한다는 해설 그대로).
- 곡선 모양 자체도 다르다: exp02는 매끈하게 오르는데 exp02b는 test_acc가
  49.7% → 46.3% → 55.0%로 **출렁이고** test_loss도 1.83 → 2.06 → 1.69로 널뛴다.
  걸음이 너무 커서 골짜기 바닥을 사이에 두고 왔다 갔다 하는 모습.
- lr 하나(10배)가 최종 성능 20.7%p를 갈랐다. "학습률 = 사전학습 가중치를 지키는
  안전장치"를 곡선으로 확인 → [q03](../ch1/q03-full-finetune-low-lr.md) ✅

## 문제가 생기면

- `CUDA out of memory` → `--batch-size 16` 추가
  ([concepts.md의 GPU 메모리 항목](concepts.md#gpu-메모리와-배치-크기))
- 에폭당 시간이 너무 길면 → `--epochs 5`로 먼저 경향만 확인
- 곡선이 이상하면 → 실험 이름 새로 지어 다시 (대시보드에 겹쳐 보며 원인 추적,
  지저분해진 실험은 `python src/metrics_logger.py remove <실험명>`)
